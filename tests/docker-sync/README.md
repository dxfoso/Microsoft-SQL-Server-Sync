# Docker SQL Sync Scenarios

This harness validates database synchronization semantics before publishing a
Windows client. It runs three independent client databases inside an isolated
SQL Server 2022 Developer container and executes SQL generated by the production
`sync_windows_agent/lib/sql_sync_merge.dart` implementation.

Run from the repository root:

```powershell
.\tests\docker-sync\run.ps1
```

Covered scenarios:

- insert, update, primary-key change, and delete
- alternate unique-key matching
- newest-database-commit conflict selection
- Arabic and Unicode values
- exact Arabic, emoji, and CJK UTF-16 storage
- null, binary, decimal, and datetime fidelity
- empty deltas and deleting an already-missing row
- independent changes from multiple writers
- an offline client catching up from an accumulated delta
- 1,200-row batched synchronization
- idempotent retry
- transaction rollback and recovery after a rejected row
- Change Tracking origin context that prevents sync echo loops

The harness is destructive only to its Docker volume. It does not connect to
the live server or live clients.

## Optional c2 Copy

The current Windows client does not provide remote database backup access.
Run this locally on the c2 database machine when a production-shaped copy is
needed:

```powershell
.\scripts\export_sync_test_database.ps1 -Database AmnDb028
```

The command creates and verifies a `COPY_ONLY` compressed backup in SQL
Server's default backup directory. Database backups can contain sensitive data:
keep them outside Git and use only an approved, access-controlled test system.

The default automated scenarios use synthetic production-like data. Importing a
real backup into the harness must remain an explicit operation because restoring
production data automatically would be unsafe.
